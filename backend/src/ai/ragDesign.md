# Chan-i 레시피 RAG 설계 개요

## 1. 코퍼스 구조

- 기본 단위: 레시피 1건
- 인덱싱용 텍스트 청크:
  - 제목: `title`
  - 재료 요약: `ingredientsSummary` (이름 + 양 + 필수/선택 여부를 자연어로 변환)
  - 조리 단계 요약: `stepsSummary` (각 step의 핵심 문장)
  - 메타데이터: 카테고리, 도구, 난이도, 칼로리, 태그

예시 JSON 스키마:

```json
{
  "id": 1,
  "title": "김치볶음밥",
  "category": "한식",
  "ingredients": [
    { "name": "밥", "amount": "2공기", "required": true },
    { "name": "김치", "amount": "150g", "required": true }
  ],
  "steps": [
    { "text": "김치를 잘게 썰어 볶는다.", "tip": "신김치 사용 시 설탕 조금 추가" }
  ],
  "tools": ["프라이팬"],
  "calories": 450,
  "difficulty": 2,
  "tags": ["자취", "한그릇", "맵게"]
}
```

벡터 인덱싱용 결합 텍스트:

```
제목: 김치볶음밥
카테고리: 한식, 태그: 자취, 한그릇, 맵게
필수 재료: 밥(2공기), 김치(150g)
선택 재료: 대파, 참기름, 김가루
조리 요약: 김치를 잘게 썰어 충분히 볶은 뒤 밥을 넣어 함께 볶는다. 마지막에 참기름과 김가루를 올린다.
사용 도구: 프라이팬
```

이 텍스트 1개를 임베딩하여 `recipe_embeddings` 테이블 또는 외부 벡터 DB에 저장합니다.

## 2. 임베딩 및 벡터 저장

- 임베딩 모델: 클라우드 LLM 제공 Embeddings API(예: text-embedding 계열) 또는 호환 오픈소스 모델.
- 저장 위치:
  - 1안: PostgreSQL + `pgvector` 확장
    - 테이블 예시:
      - `recipe_embeddings(recipe_id INT PRIMARY KEY REFERENCES recipes(id), embedding vector(1536))`
  - 2안: 외부 벡터 DB(Pinecone, Qdrant 등) 사용

### 인덱싱 파이프라인 (오프라인/관리자용)

1. `recipes` 테이블에서 레시피들을 배치로 읽어옴.
2. 각 레시피를 위 코퍼스 스키마에 맞게 직렬화.
3. 청크 텍스트를 생성하여 임베딩 API 호출.
4. 결과 벡터를 `recipe_embeddings` 또는 외부 벡터 인덱스에 upsert.

## 3. 질의 단계: 인벤토리 기반 추천

### 3.1 쿼리 빌드

입력:

- 사용자 인벤토리 스냅샷:
  - `[{ name, location, processing_state, expiry_date }, ...]`
  - 예: 삼겹살(냉장, 유통기한 임박), 김치, 달걀, 양파 등

쿼리 텍스트 구성:

```
사용자 냉장고 재료:
- 삼겹살 (냉장, D-2)
- 김치 (냉장)
- 달걀 (냉장)
- 양파 (실온)

요청:
이 재료들을 최대한 활용해서 만들 수 있는 레시피를 추천.
유통기한이 임박한 재료를 우선 사용하는 레시피를 가중치 높게.
자취생 기준, 조리 시간 30분 이하를 선호.
```

이 쿼리 텍스트를 임베딩하여 벡터 검색을 수행합니다.

### 3.2 벡터 검색

- `recipe_embeddings`에서 쿼리 벡터와의 코사인 유사도 기준 Top-K(예: 10개)를 검색.
- 검색 결과와 함께 원본 메타데이터(`recipes` + `recipe_ingredients`)를 조인.

### 3.3 LLM 재랭킹 및 설명 생성

Top-K 레시피와 인벤토리 스냅샷을 LLM에 컨텍스트로 제공하고, 다음과 같은 형식으로 응답을 생성:

프롬프트 요약:

- 컨텍스트:
  - 사용자 인벤토리 목록 + 유통기한 정보.
  - 벡터 검색으로 찾은 후보 레시피 목록(제목, 카테고리, 재료, 조리 시간).
- 지시사항:
  - 인벤토리를 최대한 활용하는 순으로 3~5개 추천.
  - 각 추천에 대해 “주요 사용 재료 / 부족한 재료 / 이 레시피를 고른 이유”를 짧게 설명.

응답 JSON 예시:

```json
{
  "recommendations": [
    {
      "recipeId": 1,
      "score": 0.92,
      "reason": "유통기한이 임박한 삼겹살과 김치를 모두 사용합니다.",
      "missingIngredients": ["두부"],
      "usesExpiringItems": ["삼겹살"],
      "estimatedDifficulty": "easy"
    }
  ]
}
```

이 JSON을 `/api/ai/recommendations` 응답 형식으로 사용합니다.

## 4. 대체 재료 및 조리법 변형

입력:

- `recipeId`
- 사용자의 인벤토리 스냅샷
- 특정 재료에 대한 “없음/대체 원함” 플래그

절차:

1. 원본 레시피 및 재료 목록을 DB에서 조회.
2. 없는 재료, 특히 필수 재료 목록을 계산.
3. LLM 프롬프트:
   - 컨텍스트: 원본 레시피 전체(타이틀, 재료, 단계, 카테고리 등).
   - 유저 인벤토리 목록.
   - 지시사항:
     - 없는 재료를 유저 재료로 대체할 수 있으면 제안.
     - 대체가 어렵다면 “생략 가능 여부”와 맛/식감 영향 설명.
     - 대체/생략을 반영한 새로운 조리 단계를 순서대로 제시.

LLM 응답을 구조화된 JSON으로 요구:

```json
{
  "adaptedIngredients": [
    { "original": "대파", "replacement": "양파", "note": "향은 비슷하지만 단맛이 더 강조됩니다." }
  ],
  "removedIngredients": [
    { "name": "참기름", "reason": "고소함이 줄지만 전체 조리에는 큰 문제 없습니다." }
  ],
  "updatedSteps": [
    "1. 양파를 잘게 썰어 파기름 대신 사용합니다.",
    "2. 나머지 과정은 원래 레시피와 동일합니다."
  ]
}
```

이 구조를 `/api/ai/substitutions` 응답 본문으로 사용합니다.

## 5. 레시피 Q&A (요리 가이드)

입력:

- 자연어 질문: 예) “이 레시피를 에어프라이어로도 만들 수 있어?”
- 현재 보고 있는 레시피의 ID와 메타데이터

프롬프트:

- 컨텍스트: 레시피 전체 정보, 사용자의 인벤토리(선택).
- 지시사항:
  - 가능한 경우, 안전성과 조리 시간을 우선으로 현실적인 조언 제공.
  - 정량적인 시간이 필요한 경우, 범위를 제시(예: 170~180도 10~15분).
  - 매우 위험한 조리법(실온 보관 육류 장시간 방치 등)은 경고 메시지 포함.

응답 구조:

```json
{
  "answer": "에어프라이어로도 가능합니다. 삼겹살을 180도로 10~12분 정도 조리한 뒤, 상태를 보고 3~5분 더 구워주세요. 중간에 한 번 뒤집어주면 골고루 익습니다.",
  "safetyNotes": [
    "기름이 많이 튈 수 있으니, 에어프라이어 바스켓에 종이호일을 깔아주세요."
  ]
}
```

이 구조를 `/api/ai/qa` 응답으로 사용합니다.
